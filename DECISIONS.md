# DECISIONS.md

Resolved architecture and design decisions for the Public Recreation Facility Management Platform backend. Per AIRules.md's Decision Velocity Rule, an entry here is settled — do not re-litigate it without the project owner explicitly asking to revisit it.

Format: **Decision** → **Choice** — *reason*. Status shows whether this is ready to build against (`Confirmed`) or needs a one-line owner/legal confirmation before being treated as final (`Needs confirmation`).

---

## Phase 1 — Architecture

**Module boundaries** → Modular monolith, one deployable, folder-per-domain: `auth`, `facilities`, `slot-booking`, `entrance-ticketing`, `payments`, `qr`, `notifications`, `admin-analytics`, `sync`, `audit-logs`. Internal contracts as TypeScript interfaces between folders. — *Avoids microservices operational overhead at solo-build scale while keeping future service extraction possible without a rewrite. Folder list harmonized with `NamingConventions.md` §3 in Phase 2 — `admin` and `analytics` are co-located as `admin-analytics` (the admin dashboard's primary job is analytics; no clean seam between them), `staff-sync` shortened to `sync`, and `audit-logs` is its own folder so the immutable-append-only DB role constraint lives next to the code that writes there.* **Confirmed.**

**Payable polymorphic contract** → `payments.payable_type` enum (`slot_reservation`, `entrance_ticket`, `shared_participant_payment`) + `payable_id` UUID. Existence/ownership validated in the service layer, not a DB foreign key (Postgres doesn't support polymorphic FKs natively). — *Standard pattern for one payment system serving multiple domains.* **Confirmed.**

**QR token design** → Signed token contains only the QR record ID, never booking data. Server looks up the record on scan and checks/sets a `used` flag. — *Signature blocks forgery; server-side used-flag blocks replay; keeping booking data out of the payload avoids leaking facility/time info if a QR image is intercepted.* **Confirmed.**

**Offline staff sync design** → Daily bulk-pull bundle scoped to facility + date, versioned by a sync timestamp; local SQLite cache on device; push via a batched endpoint, idempotency key = `device_id + qr_id + scan_time_rounded_to_minute`. — *This is the established pattern from offline ticketing/retail systems; no need to invent a new one.* **Confirmed.**

**Sync conflict resolution** → Server is the source of truth. First valid scan per QR wins. Later duplicate or conflicting scans are logged as rejected events, never silently dropped. — *Keeps a complete audit trail while still preventing double check-in.* **Confirmed.**

**Webhook idempotency** → Dedicated `webhook_events` table with a unique constraint on the provider's event ID. Processing happens inside a transaction with a row lock on the related payment record. — *Standard idempotent-webhook pattern; works regardless of delivery order or duplication.* **Confirmed.**

**Entrance capacity locking** → Atomic conditional update: `UPDATE facility_capacity SET sold_count = sold_count + :qty WHERE facility_id = :id AND date = :date AND sold_count + :qty <= max_capacity RETURNING id`. No row returned = treat as sold out. — *Avoids the lock-wait contention of `SELECT FOR UPDATE` under concurrent purchases; single round trip.* **Confirmed.**

**Audit log writes** → Dedicated `audit_logs` table. Application DB role has INSERT only — no UPDATE or DELETE grants. Partitioned by month from the first migration. — *Immutability and partitioning are far cheaper to build in now than to retrofit after the table has real volume.* **Confirmed.**

**API versioning** → URL path versioning, `/api/v1/...`. — *Simplest scheme for three independently-released clients (resident app, staff app, admin dashboard) to reason about during rollout.* **Confirmed.**

**API error response shape** → Fixed envelope: `{ error: { code, message, details } }`. `code` is a stable string enum, not just an HTTP status. — *Lets clients branch on a stable identifier regardless of wording or locale changes.* **Confirmed.**

**Rate limiting store** → Redis-backed limiter. Separate tiers: IP-based for public/auth endpoints, user-based for booking/payment endpoints, a stricter dedicated bucket for OTP requests. — *A shared store is required for limits to work once there's more than one API instance; reuses the Redis dependency already needed for queues.* **Confirmed.**

**Deployment target (initial)** → Containerized stateless API + worker processes; single managed Postgres instance + Redis instance to start (VPS or managed provider); scale out later as load requires. — *Avoids premature infrastructure complexity for a pilot-stage government system; this is a starting point, not a ceiling — Phase 1's horizontal scaling and DB scaling requirements still apply when the time comes.* **Confirmed.**

**Background job framework** → BullMQ on Redis for payment expiry, notifications, reconciliation, and sync processing. — *Mature, well-documented in the Node/TypeScript ecosystem; introduces no new infrastructure beyond the Redis instance already required.* **Confirmed.**

**Observability stack** → Structured JSON logging (pino) with correlation IDs; hosted error tracking / APM (e.g. Sentry or equivalent) rather than a self-hosted metrics stack. — *Gives real incident visibility without the operational burden of running Prometheus/Grafana for a small team.* **Confirmed.**

**Backup / disaster recovery** → Daily automated backups with point-in-time recovery via the managed database provider, 35-day retention, encrypted at rest, quarterly restore drill. — *Managed-provider PITR absorbs most of the operational complexity here; quarterly drills keep the RTO/RPO promise honest.* **Confirmed.**

**OWASP Top 10 baseline controls** → Standard layered defaults, not a custom security framework: parameterized queries via Drizzle (A03 Injection), authenticated-by-default routing with explicit public-endpoint allowlist (A01 Broken Access Control), Argon2id for any password hashing plus hashed OTPs (A02 Cryptographic Failures, A07 Identification/Auth), Zod schema validation at every API boundary including unknown-field rejection (A03, A04 Insecure Design), Helmet + strict CORS allowlist + same-site cookies on admin session paths (A05 Misconfiguration, A01), `npm audit` + Dependabot + container image scanning in CI (A06 Vulnerable Components), pino-redacted structured logging with correlation IDs and no OTP/QR-secret/token logging (A09 Logging Failures), outbound SSRF blocked by allowlisting Telebirr endpoint hosts (A10 SSRF), CSRF tokens on admin browser-session state changes (A01). Each item gets its own checklist task in Phase 2 / Phase 4 — this decision just names the baseline so individual items don't re-debate the approach. — *OWASP Top 10 has a well-established default control set for a Node/TypeScript/Postgres stack; the question isn't which framework to invent, it's which proven libraries to wire in.* **Confirmed.**

**Horizontal scaling posture** → Stateless API containers with zero in-process session/cache/lock state — all such state lives in Postgres or Redis. JWT-based access tokens (stateless verification); refresh-token records in Postgres for revocation. Rate-limit counters, BullMQ queues, and any short-lived hold/cache TTLs in Redis. Background workers run as separate container processes scaled independently from API containers. No filesystem writes for state — object storage handles uploads. Cron-style jobs run via BullMQ repeatable jobs, not node-cron in a single API process. — *Standard 12-factor posture; the only way the "scale out later" clause in the deployment-target decision is honest is if every component is written stateless from day one.* **Confirmed.**

**Database scaling path** → Single managed Postgres primary at launch with PgBouncer (transaction pooling mode) in front of it; all application connections go through the pooler, never direct. High-growth tables (`audit_logs`, `check_ins`, `webhook_events`, `notifications`, `analytics_events`, `sync_queue_entries`) are partitioned by month from the first migration so partition-level retention/archival is possible without a later schema rewrite. Read replicas are added when admin analytics or facility-discovery read load measurably impacts write latency — not preemptively. Replicas are never used for payment confirmation, capacity locking, QR validation, or any conflict-sensitive booking write (any code path touching those reads from the primary). Archival policy for partitioned tables is defined alongside the retention period (currently blocked on the legal/retention answer). — *PgBouncer + month-partitioning on known-hot tables is the proven default for a Postgres workload at this shape; adds nothing operationally expensive upfront, and keeps every later option (replicas, partition drops, archival to cold storage) open.* **Confirmed.**

**Data retention period for payments, audit logs, and personal data** → Not yet defaulted. **Needs confirmation** — this is a legal/recordkeeping question under Ethiopian financial regulation and the city's own records policy, not an engineering default. Resolve with whoever is handling the government partnership/MOU before Phase 3 schema work finalizes retention/archival columns.

**Primary key type** → UUID (preferably UUIDv7 where the runtime/library supports it) on every table; serial integer IDs are not used anywhere. — *Avoids ID enumeration on public-facing resources (facility IDs, ticket IDs), and avoids cross-environment ID collisions between dev/staging/prod seed data.* **Confirmed.**

**Polymorphic pair inventory** → Six approved pairs, each scoped to a specific purpose, no others to be introduced without logging here first: `payable_type/payable_id` (payments → slot reservations, entrance tickets, shared participant payments), `scannable_type/scannable_id` (QR codes → slot reservations, entrance tickets), `notifiable_type/notifiable_id` (notifications → any user-facing entity), `target_type/target_id` (audit logs → the entity being acted upon), `actor_type/actor_id` (audit logs → who performed the action; resident, staff, admin, or system), `aggregate_type/aggregate_id` (outbox events → the entity whose state change triggered the side effect). — *Six narrow, purpose-scoped pairs are safer than one generic polymorphic pattern reused everywhere — each pair's allowed type values are independently constrained, so a bug in QR scanning logic can't accidentally validate against an audit-log target.* **Confirmed.**

**Sync batches vs. sync queue entries** → Two separate tables: `sync_batches` (one row per pull/push operation, tracks overall batch status) and `sync_queue_entries` (one row per individual check-in/record within a batch, tracks per-record status). Batch status is derived from entry statuses, never set independently. — *Matches the original Phase 3 checklist intent ("model sync batches and sync queue records") and gives troubleshooting a natural drill-down path: see a failed batch, see which entries within it failed and why.* **Confirmed.**

**Analytics events vs. analytics reports** → Two distinct concepts, not one: `analytics_events` is an internal write-side table capturing raw operational events (bookings, payments, check-ins) as they happen; `AnalyticsReport` is a read-side type computed via background aggregation jobs or materialized views, exposed at `/v1/analytics-reports`. Raw events are never returned directly from the reports endpoint. — *Keeps the high-volume write path (every booking/payment/check-in writes an event) decoupled from the read path the admin dashboard actually queries, which needs to stay fast regardless of how large the event table grows.* **Confirmed.**

**HTTP framework** → NestJS. — *The naming conventions already lock in a class-based, suffix-driven style (`Controller`, `Service`, `Repository`, `Policy`, `Mapper`, `Validator`) and a folder-per-domain module structure (`auth`, `facilities`, `slot-booking`, etc.). NestJS's module/controller/service/DI system maps directly onto both of those without fighting the framework, and its built-in guards and pipes give RBAC and request validation a natural home.* **Confirmed.**

**ORM / query layer** → Drizzle ORM. — *This system leans heavily on hand-written SQL for the parts that matter most: atomic conditional updates for capacity locking, exclusion constraints for slot-overlap prevention, and partial indexes. Drizzle stays close to SQL and makes mixing typed queries with raw SQL low-friction; Prisma fights harder against exclusion constraints and conditional atomic updates, which are exactly the patterns this system depends on for correctness.* **Confirmed.**

**Validation library** → Zod. — *Integrates directly with NestJS pipes for request validation at the API boundary; widely adopted, well-typed, no surprises.* **Confirmed.**

**Test framework** → Vitest. — *Faster and more native to ESM/TypeScript than Jest, with a near-identical API so nothing is lost; matters for a test suite this large given the Testing Gate requires a real test per Critical-priority item.* **Confirmed.**

## Phase 7 — Payments

**Telebirr refund mechanics** → **Needs confirmation.** Community SDKs (e.g. the `telebirr-php` wrapper) expose a `refundOrder` call taking the original payment reference and amount, suggesting programmatic refunds are plausible. This has not been verified against Ethio Telecom's own merchant documentation. Confirm during merchant onboarding: whether refunds are supported via API at all, minimum/maximum amounts, time-window restrictions after the original transaction, and whether refunds need manual approval on Telebirr's side. Until confirmed, treat refund-related Phase 7 checklist items as "build the request/approval flow, but the final settlement step may need a manual ops fallback rather than a pure API call." **Update:** the developer portal's Payment Details flow includes a step labeled "Explore other Payment API" after the core integration steps — this is the most likely place refund/other endpoints are documented. Still not confirmed; check there directly once through registration.

**Telebirr authentication and signing mechanism** → Confirmed directly from the official Node.js demo, C2B WebCheckout demo, and SignTool. Two independent layers, both required: (1) **Gateway access token** — exchange `appId` + `appSecret` for a short-lived bearer token via `POST /payment/v1/token`; sent as the `Authorization` header on every call. (2) **Per-request RSA-PSS signature** — every request body is signed with the merchant's own RSA private key, algorithm `SHA256withRSAandMGF1` (RSA-PSS, not PKCS#1v1.5). Canonicalization: flatten top-level fields plus everything inside `biz_content`, exclude `sign`/`sign_type`/`header`/`refund_info`/`openType`/`raw_request`/`biz_content` itself, sort keys alphabetically (ASCII), join as `key=value&key=value`, RSA-PSS-SHA256-sign with the merchant private key, base64-encode as the `sign` field; `sign_type` is sent as `"SHA256WithRSA"`. The "Sign Tool" from the portal is a Java/Bouncy Castle GUI for generating this RSA key pair during merchant onboarding — the public key gets registered with Ethio Telecom, the private key stays on the server and is used for every outbound signature. Node's built-in `crypto.sign` supports RSA-PSS natively (`RSA_PKCS1_PSS_PADDING`, Node 15+) — no need to port the bundled `jsrsasign` library into the real adapter. — *This is the actual mechanism, not a planning assumption — verified by reading the official SDK code directly.* **Confirmed.**

**C2B web checkout flow shape** → Confirmed: order creation is `POST /payment/v1/merchant/preOrder` with `trade_type: "Checkout"` (the mini-app embedded flow uses `"InApp"` instead — not used by this system). The response includes a `prepay_id`; the backend then builds a signed checkout URL (same signing scheme, different field set) that the resident's browser/app opens directly — that URL is Telebirr's own hosted payment page, not something this system renders. `notify_url` is a field supplied at order-creation time — Telebirr calls it asynchronously on payment completion, confirming the webhook design already locked in above. No sample webhook receiver or inbound-signature-verification code is included in any of the three official demo packages reviewed (Node.js demo, C2B WebCheckout demo, SignTool) — that detail still needs the actual API reference doc (likely under "Explore other Payment API" on the developer portal), not assumed by symmetry with outbound signing. **Confirmed (request flow); webhook verification spec still needs the API reference doc.**

**Telebirr refund/query/cancel endpoints** → **Still not confirmed — checked and not found.** None of the three official demo packages (Node.js demo, C2B WebCheckout demo, SignTool) contain a refund, order-cancellation, or order-query endpoint anywhere in the code. `refund_info` appears only as an excluded-from-signature field name in the shared signing utility, with no corresponding service that populates or sends it. This is stronger evidence than before that refunds, if supported at all, live in a separate part of the API reference not covered by these starter demos — check the actual PDF/reference doc under "Explore other Payment API" directly rather than inferring from code.

**Sandbox credential hygiene** → The official `C2B_WebCheckoutDemo` ships a real-looking RSA private key, `appId`, `appSecret`, and merchant code hardcoded in `config.js`, in plaintext. Treat this as a shared/sample sandbox credential for getting the demo running locally only — never reuse it in the actual codebase, never commit it to the project repo, and replace it with your own credentials and key pair (generated via SignTool) once your own merchant account is approved. — *Matches the existing secrets-management rule in AIRules.md; flagging explicitly since the official demo itself doesn't follow that practice.* **Confirmed.**

**Payment provider integration timing** → Build behind a `PaymentProvider` interface starting in Phase 6. A mock/sandbox provider (deterministic success, failure, and delay responses) is implemented first and used for all Phase 6 business-logic work and Phase 9 testing. The real Telebirr adapter — signature verification, live webhook handling, refund calls — is built in Phase 7 once merchant account access and official documentation are available, and swaps in behind the same interface without changing anything upstream. — *Telebirr merchant onboarding is an external business-side dependency outside engineering control and can take weeks; the rest of the backend (slot booking, entrance ticketing, QR issuance, capacity locking) should not be blocked waiting on it. This also makes explicit, from Phase 6 onward, what Phase 9 already assumed by listing a "fake Telebirr provider" as a test dependency — the mock isn't a testing afterthought, it's the thing the whole payment-dependent system is built against until Telebirr is actually live.* **Confirmed.**

## Ambiguity protocol

If a new decision is needed that doesn't fit an existing entry:
1. Check if it's security/payment/capacity/legal-critical (see AIRules.md Decision Velocity Rule exception). If yes and genuinely ambiguous, escalate to the project owner instead of deciding alone.
2. Otherwise, pick the established default for this stack, record it here with the one-line reason, and move on. Don't leave it open while continuing to discuss it.
