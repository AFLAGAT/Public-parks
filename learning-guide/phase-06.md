## Implement payment provider abstraction with a mock/sandbox adapter

**What was done.** Introduced the payments domain's provider boundary — a single `PaymentProvider` interface (`src/payments/payment-provider.port.ts`, DI token `PAYMENT_PROVIDER`) plus a real, deterministic `MockPaymentProvider` implementation (`src/payments/mock-payment.provider.ts`) — and bound the token to the mock in `PaymentsModule`. Provider-agnostic contract types live in `payment-provider.types.ts`. This is the first Phase 6 item and the enabler for the active slice's `entrance ticket → mock payment → QR issuance` segment: everything payment-dependent is now built and tested against this interface instead of against Telebirr.

**Why.** DECISIONS.md ("Payment provider integration timing", Confirmed) settled mock-first: Telebirr merchant onboarding is an external, weeks-long business dependency, so the backend is built behind a `PaymentProvider` interface and the real Telebirr adapter swaps in behind the *same* contract in Phase 7, changing nothing upstream. The mock isn't a testing afterthought — it is the thing the whole payment-dependent system is built against until Telebirr is live.

**How it works.**
- `initiatePayment(request)` opens a payment and resolves to `initiated` (with `prepayId` + provider-hosted `checkoutUrl`) or a classified `failed`. It is **success-or-fail, never `confirmed`** — confirmation is always asynchronous, faithfully modeling Telebirr's `POST /preOrder` → `prepay_id` → hosted checkout → async `notify_url` flow (DECISIONS.md "C2B web checkout flow shape"). The mock generates a `prepayId`/`checkoutUrl` by default and can be programmed per-order to fail via `programInitiationFailure(...)`.
- `verifyCallback(rawPayload)` is a **pure, synchronous** function that verifies and normalizes an inbound provider callback into a discriminated result (`verified` | `signature_invalid` | `unrecognized`). Signature verification lives in the provider because only the provider knows its signing scheme (the mock uses an HMAC-SHA256 over canonical fields as a stand-in for the real adapter's RSA-PSS verification per DECISIONS.md "Telebirr authentication and signing mechanism"). On `verified` it surfaces `merchantReference`, `providerTransactionId`, `paymentOutcome`, and the provider-reported `amount` (surfaced, never trusted — the caller compares it to the expected amount). Every result carries a `redactedPayload` with provider-secret fields stripped, ready for the append-only `webhook_events` log.
- `buildCallback(...)` is a sandbox affordance that produces a correctly-signed mock callback, simulating the provider calling our `notify_url`; downstream Phase 6 work and tests use it to drive `verified` callbacks.

**Key concepts / why the shape.** The contract carries only what the payments domain needs (integer-santim money, our `merchantReference`, the provider's txn id, a normalized outcome) — no Telebirr-specific request/response shapes leak across the boundary. `verifyCallback` deliberately **cannot** return `duplicate`: exactly-once is enforced downstream by the `processed_provider_events` ledger (DECISIONS.md), so the provider's only job is authenticity + interpretation. Purity is the critical-path property — the same callback verified twice yields identical results, so duplicate/out-of-order deliveries never double-confirm at the provider layer; ordering/dedup is the ledger's concern.

**Testing (critical-path failure-mode gate).** Pure logic, no DB touch, so this is a unit-tier item (the DB-enforced payment invariants were covered at the Phase 3 modeling items). Failure modes exercised: programmed initiation failure returns a classified `failed` result **without throwing**; a callback with a **tampered amount** breaks the signature (`signature_invalid`, never a confirmable result — money-tamper detection); a forged signature is rejected; a payload missing/with a non-integer `amount` is `unrecognized`; and repeated (duplicate/out-of-order) verification is deterministic. 13 new unit tests; full suite 206/206, typecheck, lint, and production build green.

**Pitfalls / mistakes to avoid.**
- Do **not** let `initiatePayment` return a confirmed/paid state — Telebirr confirmation is async; treating initiation as confirmation would skip webhook verification entirely.
- Do **not** move dedup/idempotency into `verifyCallback`; that belongs to the `processed_provider_events` ledger. Keeping the provider pure is what makes replay handling correct.
- Always compare the callback's reported `amount` to the expected amount at the call site — the provider surfaces it but does not (and should not) know what was expected.
- The mock HMAC secret is development/test-only; the Phase 7 Telebirr adapter verifies RSA-PSS with the merchant key and must not reuse any sandbox credential (DECISIONS.md "Sandbox credential hygiene").

**Deviation.** None beyond scope discipline: refund is intentionally left off the interface for now — the active slice does not refund, and Telebirr refund mechanics are still unconfirmed (DECISIONS.md "Telebirr refund mechanics/endpoints"). Adding a `refundPayment` method later is an additive, non-breaking change to both adapters, to be done when the refund slice is built.
