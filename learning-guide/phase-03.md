# Phase 3 — Database Design — Learning Guide

Append-only build log entries for Phase 03 checklist items. New entries go at the bottom.

## Model users with client-neutral identity

- **What was done:** Added the first domain table, `users`, to `src/database/drizzle.schema.ts` and generated `drizzle/0001_create_users.sql` plus its Drizzle snapshot/journal entry. A user has a database-generated UUID primary key, optional phone and email identity channels, per-channel verification timestamps, `is_active`, `created_at`, and `updated_at`. Named unique indexes protect phone/email identity, while named checks require at least one channel, enforce E.164 phone formatting and lowercase/trimmed email normalization, and forbid verification state without the corresponding identifier. The table deliberately has no role, client type, password, OTP, or reset-token column.

- **Why:** Resident, staff, and admin applications need to refer to the same human identity without copying accounts or baking a current role/client into the identity row. Keeping credentials and authorization out of `users` lets later authentication methods and role assignments evolve independently, while the UUID remains the stable owner/actor reference across bookings, payments, QR checks, and audits. Avoiding credential columns at this stage also makes it impossible for this schema item to accidentally introduce plaintext OTP or reset-token storage.

- **How it works:** PostgreSQL generates `users.id` with `gen_random_uuid()` and default lifecycle values. Either `phone_number` or `email` must be present; both may be present so one identity can gain another verified login channel without creating a second user. Unique indexes reject duplicate non-null identifiers. Check constraints enforce normalized storage even if a future caller bypasses API validation, and verification timestamps cannot exist for absent channels. The future Phase 4 authentication layer will validate and normalize external input before insert, store credentials in their owning model using approved hashing, and resolve a verified channel back to this stable user ID.

- **Key concepts:** Client-neutral identity; separation of identity, credentials, and authorization; nullable unique identity channels; defense-in-depth database constraints; E.164 phone normalization; verification state tied to its source identifier; generated UUID primary keys.

- **Best practices applied:** Every constraint and index follows the binding naming convention. Identity formats and uniqueness are database-enforced rather than relying only on future controller validation. Verification timestamps record state without storing OTP values. Integration tests apply both migrations to disposable PostgreSQL, accept valid phone/email users, and exercise duplicate, missing, malformed, unnormalized, and inconsistent verification failure modes. The test run passed 122 unit and 17 integration tests plus typecheck, lint, and production build.

- **Mistakes to avoid:** Adding `role`, `user_type`, or `client_type` to `users`; creating separate resident/staff/admin identity tables; storing a plaintext password, OTP, or reset token; accepting local-format phone numbers without canonical E.164 normalization; setting a verification timestamp when its channel is null; relying on application-only uniqueness; or reading PostgreSQL error codes directly from Drizzle's wrapper instead of its nested `cause`. Drizzle's `_journal.json` records migration order and tags but not SQL hashes—the applied hash must be checked against SHA-256 of the migration file itself.

## Model facilities with operational classification

- **What was done:** Added `facility_types`, the `facility_operational_classification` enum (`slot_based`, `entrance_based`), and `facilities` in migration `0002_create_facilities.sql`. Each facility references exactly one type and stores normalized public name/address fields, active state, optional description, timestamps, and a PostGIS `geometry(Point,4326)` location; a GiST expression index on `location::geography` supports distance searches in meters. The new decision is recorded in `DECISIONS.md` so later courts, capacity, and discovery code share one classification source.

- **Why and verification:** Keeping classification on the reference type prevents individual facilities from drifting into an impossible type/mode combination, while the typed point and geography index make nearby discovery accurate and indexable. Tests against disposable PostGIS cover both classifications, EWKB read/write mapping, a real two-kilometer query, index definition, invalid enum/type codes, blank public fields, invalid coordinates, unknown types, and deletion of referenced types; 125 unit and 23 integration tests plus typecheck, lint, and build pass.

- **Non-obvious gotchas:** Drizzle 0.45 accepts an `srid` option on its geometry type but currently emits plain `geometry(point)`, so this schema uses a typed custom column that explicitly generates `geometry(Point,4326)` and delegates EWKB parsing to Drizzle's exported parser. Migration generation also revealed that the hand-written `0000_snapshot.json` pointed to itself; assigning it a real snapshot ID and linking `0001` to it restored the linear `0000 → 0001 → 0002` chain without changing any applied SQL migration.

## Model authentication, SMS revisions, and immutable audit records

- **What was done:** Added roles/permissions, credentials, encrypted TOTP factors, one-use recovery codes, client-typed sessions, hashed refresh tokens, digest-only OTP challenges, scoped SMS configuration revisions/tests/delivery attempts, and monthly partitioned audit logs. Platform scope is enforced as a null `scope_id`; city scope is modeled but requires a non-null ID and is not exposed by service APIs.
- **Why:** These records keep identity, authentication method, authorization, delivery infrastructure, and accountability independent. Scope-aware uniqueness and activation constraints avoid a table redesign when Super Admin-managed city overrides arrive.
- **How it works:** SMS rows carry scope, provider, revision, encrypted credentials, exact-revision test state, and lifecycle actors/timestamps. A partial unique index permits one active row per scope. Audit events append actor/action/target/correlation/sanitized metadata into monthly partitions; row and statement triggers reject UPDATE, DELETE, and TRUNCATE. Deployment applies SELECT/INSERT-only grants to a non-owner runtime role.
- **Key concepts:** Credential separation; scope-ready fallback design; immutable revisions; database-enforced singleton activation; append-only partitioned accountability records; migration-owner/runtime-role separation.
- **Best practices applied:** Foreign keys protect configuration actors, check constraints bound timeouts/retries/scope shape, sensitive destinations are hashed or masked, and activation plus its audit event share one transaction. Fresh PostgreSQL tests prove seeded RBAC, city-scope rejection, one-active-provider enforcement, and mutation/truncate rejection.
- **Mistakes to avoid:** Making the runtime role the owner of `audit_logs` (owners retain implicit privileges); putting a nullable scope directly in a normal unique key without `COALESCE`; storing provider secrets as ordinary JSON; or creating future partitions without the maintenance/grant procedure. PostgreSQL requires every partition key column in a partitioned table's primary key, so `audit_logs` uses `(id, created_at)`.

## Model staff assignments by facility and time range

- **What was done:** Added `staff_assignments` (migration `0005_sharp_synch.sql`), the server-side source of truth for which staff `user_id` may act at which `facility_id`, bounded by an explicit `starts_at`/`ends_at` window. Columns: `assignment_status` (`staff_assignment_status`: `active`/`revoked`, default `active`), the validity window (`ends_at` nullable = open-ended), revocation metadata (`revoked_at`, `revoked_by_user_id`), and accountability (`assigned_by_user_id`). Four foreign keys to `users`/`facilities` are all `ON DELETE restrict`. Constraints: `chk_staff_assignments__time_range` rejects inverted/empty windows; `chk_staff_assignments__revocation_consistency` binds `revoked_at`+`revoked_by_user_id` to the `revoked` status (and forbids them on `active`); a partial unique index `uidx_staff_assignments__user_facility__where_active` allows at most one active assignment per staff+facility. Two non-unique indexes (`facility_id`+status, `user_id`+status) serve the two hot lookup directions (staff daily schedule / admin facility roster vs. per-staff scope checks).

- **Why:** Staff facility scope must be enforced by the backend, never by hidden UI. This model is the data foundation Phase 4 "Enforce staff facility scope server-side" and every staff endpoint (schedule view, QR validation, offline sync pull/push) read to answer "is this staff member authorized here, now?". Modeled ahead of the resident-facing entrance-ticket chain in the active slice because that chain (`entrance_tickets`, `facility_capacities`, `payments`) touches AIRules stop-and-ask tables and awaits owner schema confirmation, whereas staff assignments depend only on already-modeled `users`/`facilities` and are unblocked.

- **How it works:** The authorization predicate is: `assignment_status = 'active' AND starts_at <= T AND (ends_at IS NULL OR ends_at > T)` for the exact `(user_id, facility_id)`. Because the revocation-consistency check guarantees an `active` row can never carry revocation metadata, the status filter alone is sufficient to exclude revoked assignments — revoking (set status + `revoked_at` + `revoked_by_user_id`) denies future access immediately. The partial unique index enforces the "one active assignment per facility" invariant while still permitting sequential reassignment: revoke the old row, then insert a new active row.

- **Key concepts:** Server-side scope as data, not UI; validity windows with an open-ended upper bound; status-plus-window authorization predicate; revocation as immediate denial; database-enforced single-active invariant via a partial unique index; `ON DELETE restrict` to preserve historical accountability; who-assigned/who-revoked accountability columns.

- **Best practices applied:** Every constraint/index follows the naming convention. The single-active invariant and the time-range/revocation invariants are database-enforced, not left to controllers. Real-Postgres failure-mode tests (Testing Gate: critical path) prove the security-relevant denials — a revoked assignment inside its window is denied, not-yet-started and already-ended assignments are denied, and a facility the staff member is not assigned to is denied — plus inverted range rejection, revocation-inconsistency rejection, duplicate-active rejection with post-revocation reassignment, and FK protection of referenced users/facilities. 174 unit, 45 integration (8 new), typecheck, lint, and build pass.

- **Mistakes to avoid:** Enforcing facility scope only in the query without a status/window guarantee (a revoked-but-in-window row would leak access); allowing multiple active assignments per staff+facility (ambiguous scope, harder revocation); using `ON DELETE cascade` on the staff/facility FKs (would silently erase accountability history when a user or facility is removed); putting revocation metadata on `active` rows; forgetting that leftover rows in the shared integration database with `ON DELETE restrict` break sibling specs' parent-row cleanup — this spec deletes its own `staff_assignments`/`facilities`/`facility_types`/`users` rows in `afterAll`, child-first.

## Model QR codes as polymorphic access credentials

- **What was done:** Added `qr_codes` (migration `0006_slimy_greymalkin.sql`), one credential pattern that references either a slot reservation or an entrance ticket through the approved `scannable_type` / `scannable_id` polymorphic pair. `scannable_type` is the `qr_scannable_type` enum (`slot_reservation`, `entrance_ticket`); `qr_code_status` is the `qr_code_status` enum (`active`, `revoked`, default `active`); `revoked_at` is bound to the `revoked` status by `chk_qr_codes__revocation_consistency`; `created_at` is the issuance time. A partial unique index `uidx_qr_codes__scannable__where_active` permits at most one active credential per booking.

- **Why:** The platform needs one QR subsystem for two booking types without duplicating credential logic, while guaranteeing a credential can only ever point at a *scannable* entity. Modeling `scannable_type` as an enum (not a free varchar like `audit_logs.target_type`) enforces the confirmed "each polymorphic pair's allowed values are independently constrained" property at the database level — a bug elsewhere cannot mint a QR against a payment, a participant, or an audit target. Built now because it is the QR step of the active slice and, being polymorphic with no hard FK, does not depend on the entrance-ticket / slot-reservation tables (which touch stop-and-ask tables and await owner schema confirmation).

- **How it works:** Per the confirmed QR token design, the signed token contains only this row's `id`; the server verifies the signature and looks the row up on scan — so no reusable secret is stored, and there is deliberately no `token`/`secret`/`used` column. Revocation (set status + `revoked_at`) denies future scans immediately, and because the consistency check forbids revocation metadata on `active` rows, the `qr_code_status = 'active'` filter alone is a sufficient validity gate. Single-use (slot) and quantity-aware (entrance) consumption are *not* modeled as a boolean here — that would not represent multi-entry tickets — but are recorded in `check_ins`, the idempotency-keyed scan ledger, consistent with the confirmed sync-conflict rule ("first valid scan per QR wins; later scans logged as rejected events"). The one-active-per-booking partial unique index prevents a second live (replayable) credential for the same booking while still allowing a fresh credential to be issued after the previous one is revoked.

- **Key concepts:** Enum-constrained polymorphic reference; credential identity vs. consumption ledger separation (single source of truth for "used" lives in `check_ins`); signed-token-of-record-id (no stored secret); revocation as immediate denial; database-enforced single-active-credential invariant; intentional absence of a foreign key for a polymorphic reference.

- **Best practices applied:** The scannable allowlist and the active-credential invariant are database-enforced, not left to services. Real-Postgres failure-mode tests (Testing Gate: QR is a critical path) prove the security-relevant behaviors — a non-scannable type (`shared_reservation_participant`) is rejected by the enum, a revoked credential is excluded from the active-credential lookup, the one-active-per-booking invariant holds with post-revocation reissuance, revocation-state inconsistency is rejected, and the invariant is correctly scoped per `(type, id)`. 178 unit, 50 integration (5 new), typecheck, lint, and build pass.

- **Mistakes to avoid:** Storing a raw or hashed reusable QR secret when the signed-record-id design needs none; adding a `used`/`consumed_at` boolean to `qr_codes` (splits the security-critical "is this consumed?" truth between here and `check_ins`, and cannot express entrance-ticket quantity); using a free-text `scannable_type` varchar that would let a QR reference a non-scannable entity; adding a foreign key to a polymorphic column; or allowing more than one active credential per booking (a second live credential is a replay vector).

## Model check-ins with idempotency keys

- **What was done:** Added `check_ins` (migration `0007_thin_purple_man.sql`), the scan/attendance ledger written by online staff QR validation and by offline sync push. Each row captures the scan actor (`staff_user_id`), the reporting `device_id`, the `facility_id` scope, the `validation_source` (`online` / `offline_sync`), the scanned `qr_code_id`, and the outcome (`check_in_result` = `accepted` / `rejected`, with a bound `rejection_reason`), plus a nullable `sync_batch_id`, `correlation_id`, and the `scanned_at` / `scan_minute` pair. The confirmed idempotency key `device_id + qr_code_id + scan_time_rounded_to_minute` is enforced by the unique index `uidx_check_ins__idempotency (device_id, qr_code_id, scan_minute)`.

- **Why:** Offline devices resend scans; without server-side dedup a single logical scan would create multiple attendance records. The idempotency key absorbs exact replays, while genuinely distinct or conflicting scans (a different device, or a single-use QR scanned again) are inserted as separate rows and marked `accepted` or `rejected` — logged, never silently dropped, exactly as the confirmed sync-conflict rule requires. This is the check-in step of the active slice; it depends only on already-modeled `qr_codes` / `users` / `facilities`.

- **How it works:** `check_ins` is one of the confirmed month-partitioned high-growth tables, but it is partitioned by `scan_minute`, not `created_at`. The reason is a hard Postgres rule: a unique (or primary) constraint on a partitioned table must include every partition-key column. Partitioning by `created_at` would force the idempotency index to include `created_at`, and two replays inserted at different times would then both pass — dedup defeated. Partitioning by `scan_minute` (which is *part of* the idempotency key) lets the unique index include the partition key and enforce true global dedup. `scan_minute` is a plain column (Postgres forbids a generated column as a partition key) constrained by `chk_check_ins__scan_minute_truncated` to equal `date_trunc('minute', scanned_at)`. The primary key is `(id, scan_minute)`. The migration adds `PARTITION BY RANGE (scan_minute)` and current/next/default partitions (mirroring `audit_logs`); `npm run checkins:maintain-partitions` creates three months ahead so writes avoid the default partition under normal operation.

- **Key concepts:** Idempotency-key dedup for offline replay; partition-key-must-be-in-the-unique-key Postgres constraint driving the partition-column choice; truncated-minute column as both dedup granularity and partition key; accepted-vs-rejected ledger (conflicts logged, not dropped); attendance vs. rejected-scan distinction; a hand-finished partitioned migration alongside a drizzle schema that is unaware of partitioning.

- **Best practices applied:** The idempotency key, the scan-minute truncation, and the rejection-reason consistency are all database-enforced. FKs to `qr_codes` / `users` / `facilities` use `ON DELETE restrict` to preserve attendance history. Real-Postgres failure-mode tests (Testing Gate: check-in idempotency is a critical path) prove: an exact replay is rejected by the idempotency index (whose violation, on a partitioned table, surfaces against the leaf partition's index), distinct and rejected scans coexist as separate rows, a mistruncated `scan_minute` and an inconsistent `rejection_reason` are rejected, references are FK-protected, rows route to the correct monthly partition, and out-of-range scans fall to the default partition. 181 unit, 56 integration (6 new), typecheck, lint, and build pass.

- **Mistakes to avoid:** Partitioning `check_ins` by `created_at` for consistency with `audit_logs` (it silently breaks idempotency enforcement); asserting a partitioned unique violation against the parent index name (Postgres reports the auto-named leaf-partition index — assert on the key columns instead); using a generated column as the partition key (rejected by Postgres); adding a boolean "used" flag here or on `qr_codes` (consumption is this ledger); forgetting partition maintenance (writes silently pile into the default partition, losing the retention/archival benefit); or omitting the truncation CHECK (an app bug could store a `scan_minute` that doesn't match `scanned_at`, corrupting dedup).
