# Phase 4 — Authentication and Security — Learning Guide

Append-only build log entries for Phase 04 checklist items. New entries go at the bottom.

## Implement resident OTP and Super Admin MFA authentication

- **What was done:** Added canonical E.164 OTP challenges, cryptographically random six-digit codes, keyed digest storage, five-minute expiry, prior-challenge invalidation, five transactional attempts, Redis resend/request limits, resident session issuance, and rotating refresh tokens. Added Super Admin Argon2id password challenges, encrypted TOTP, hashed one-use recovery codes, replayed-time-step rejection, secure cookie refresh with Origin/CSRF validation, JWT access tokens, reuse-triggered revocation, and interactive provisioning.
- **Why:** Resident authentication remains simple on mobile while privileged browser access receives password plus MFA and stronger refresh protections. Client type, role, and explicit permission are all checked for SMS management, so a valid resident/City Admin/gate-worker token is still denied.
- **How it works:** The request path hashes phone/IP limiter keys, stores only `HMAC(challenge, phone, OTP)`, and sends through `OtpDeliveryPort`. Verification locks and consumes the challenge before creating identity/session/token records. Access guards verify JWT key ID/signature/issuer/audience and reload active session/roles/permissions from PostgreSQL. Refresh rotation locks the hashed token row; seeing a rotated token revokes its whole session. Admin MFA challenges live five minutes in Redis and only create a session after password plus non-replayed TOTP/recovery verification.
- **Key concepts:** Online brute-force budgets; keyed OTP digests; transactional consume-once credentials; short access/long rotating refresh tokens; token-family reuse response; MFA time-step replay defense; browser Origin/CSRF defense; client type as an authorization boundary.
- **Best practices applied:** Unknown admin accounts perform a dummy Argon2id check, inactive users fail closed, secrets/tokens/OTPs never enter logs or audit metadata, admin cookies are Secure/HttpOnly/SameSite=Strict, and provisioning is interactive-only. Real PostgreSQL/Redis HTTP tests cover expiry, five-attempt exhaustion, resend cooldown, delivery outage, OTP replay, refresh reuse revocation, TOTP replay, bad Origin/CSRF, and resident/City Admin/gate-worker denial.
- **Mistakes to avoid:** Looking up only active refresh tokens during reuse detection; trusting roles embedded in a long-lived JWT; marking a challenge consumed outside the session transaction; logging the development OTP; accepting local phone formats; or treating a valid City Admin token as sufficient for platform operations. OTP verification locks the challenge row so concurrent submissions cannot both create sessions.

## Implement provider-agnostic SMS security

- **What was done:** Added encrypted, write-only credentials; provider-declared credential metadata/config/host validation; exact-revision test requirements; transient-only bounded retry and timeout handling; database-per-send active resolution; development/test-only mock delivery; and a development-only, timing-safe-token-protected, bounded in-memory inbox. No OTP or provider response is logged.
- **Why:** Ethio Telecom can be integrated later as one adapter registered behind the same provider interface. Production has no mock fallback and returns the sanitized `SMS_DELIVERY_UNAVAILABLE` failure when an active provider cannot deliver.
- **How it works:** The resolver accepts optional city context but currently loads the active platform row on every send. The registry returns the implementation by stable key; the dispatcher validates host/config, supplies an abort signal and stable idempotency key, records only normalized outcomes, and retries only transient classifications. Testing uses fixed server content and marks the exact revision; activation atomically replaces the prior active row and appends its audit event.
- **Key concepts:** Dependency inversion; provider capability metadata; SSRF host allowlists; encrypted JSON credentials; authenticated encryption AAD; immutable tested revisions; transient/permanent/configuration failure classes; production fail-closed behavior.
- **Best practices applied:** Secrets are write-only, API responses expose booleans only, mock registration is absent in staging/production, and the development inbox holds at most 100 messages for ten minutes. Unit tests prove transient-only retry, permanent failure behavior, timeout classification, registry uniqueness, bounded mock storage, authenticated encryption, and future city-context compatibility.
- **Mistakes to avoid:** Copying ciphertext into a new revision (AAD binds it to the old ID/revision); retrying permanent failures; caching the active provider across sends; allowing arbitrary API hosts; exposing raw provider responses; or silently falling back to mock in production.
