## Initialize production-oriented project structure

- **What was done:** Scaffolded the NestJS project with the ten domain folders from `NamingConventions.md` §3 (`auth`, `facilities`, `slot-booking`, `entrance-ticketing`, `payments`, `qr`, `notifications`, `admin-analytics`, `sync`, `audit-logs`). Added `package.json`, strict `tsconfig.json` + `tsconfig.build.json`, `nest-cli.json`, `vitest.config.ts`, a minimal `src/main.ts` bootstrap that mounts everything under the `/v1` prefix, an `AppModule` that imports the ten domain modules, and an empty `<module>.module.ts` per folder. Added one smoke test that compiles `AppModule` via `@nestjs/testing` so Vitest has something to run. `npm run build` and `npm test` both pass on Node 26 / npm 11. Also harmonized `DECISIONS.md` so the module-boundaries entry matches the `NamingConventions.md` folder list — they had drifted (`staff-sync`/`admin`/`analytics` vs. `sync`/`admin-analytics`/`audit-logs`).
- **Why:** A flat prototype layout would make module-boundary discipline rely on memory. Putting the folders in place first means every later checklist item gets dropped into the right module by default, and reaching across module boundaries becomes visibly wrong instead of unnoticed. Bootstrapping with `/v1` already wired honors the path-versioning decision from day one, so no later "where does the v1 prefix come from" question.
- **How it works:** `nest build` (via `@nestjs/cli`) compiles `src/` to `dist/` using `tsconfig.build.json`. `npm test` runs `vitest run` against `src/**/*.spec.ts`. The smoke test calls `Test.createTestingModule({ imports: [AppModule] }).compile()` — that pulls in every domain module and verifies the DI graph wires without errors. `AppModule.imports` is the registration point for the ten domain modules; today they are empty `@Module({})` shells, but every later Phase 2/3/4 item will register controllers, providers, and repositories inside its own module without touching this list again.
- **Key concepts:** NestJS module system, dependency-injection composition, `Test.createTestingModule`, strict TypeScript (`strict`, `noUnusedLocals`, `noImplicitReturns`, etc.), CommonJS output for Node, `reflect-metadata` at the entry point for decorator metadata.
- **Best practices applied:** Strict TypeScript on from the first file, not "we'll tighten it later." Decorator metadata polyfill imported once in `main.ts` and the test file rather than scattered. URL prefix set globally instead of per-controller. Smoke test exercises real `AppModule` composition — a typecheck would also pass on an unimportable module, the smoke test would not. Module class names follow `NamingConventions.md` §3 (PascalCase, `Module` suffix). File names follow §3 kebab-case (`<name>.module.ts`).
- **Mistakes to avoid:** Adding controllers/services to the root `AppModule` "temporarily" — they then own no domain folder and never get extracted. Skipping the `bootstrap()` IIFE pattern (`void bootstrap()`) and letting `await NestFactory.create(...)` float at the top level — Node will hide the rejection. Using globs in `tsconfig.include` that pick up `dist/` or `node_modules/` — slow and produces phantom errors. Letting one module folder grow `index.ts` re-exports of internal types — the whole point of folder-per-domain is that other modules consume the typed module interface, not internal symbols. Inviting transitive-dep vulnerabilities to block this item — `npm install` reports 24 advisories (mostly from the NestJS CLI's older glob chain); these are scoped to dev-time tooling and will be handled by the Phase 4 "dependency and vulnerability scanning" item, not silently here.

## Create configuration management layer

- **What was done:** Added `src/config/` with a section-composed Zod schema (`env.schema.ts`), a typed accessor service (`AppConfigService`), and a global NestJS module (`ConfigModule`) that wires `@nestjs/config`'s `forRoot` with our Zod validator. The schema's `validateEnv` function runs at startup; if any value is missing, malformed, or out of range the app refuses to boot with a multi-line error pointing to each offending variable. `main.ts` now resolves the port through `AppConfigService` instead of reading `process.env.APP_PORT` directly. Vitest sets `APP_NODE_ENV=test` so the existing `AppModule` smoke test (which boots the real DI graph) still passes. Added `.env.example` as the canonical reference for what every variable means. New dependencies: `@nestjs/config` and `zod` — both directly implement DECISIONS.md choices (Zod is the named validation library; `@nestjs/config` is the standard config module for NestJS). 10/10 tests pass (`validateEnv` × 7, `AppConfigService` × 2, `AppModule` smoke × 1).
- **Why:** Every later Phase 2/3/4 item — database URL, Redis URL, JWT signing key, Telebirr credentials, queue settings — needs an answer to "where do I read this and what guarantees does it carry?" Putting the typed, fail-fast config layer in *before* any of those means every later item just adds a section to the schema and a getter to the service — nothing reads `process.env` directly again. Doing this Critical-priority item second (right after the skeleton) means no insecure pattern of "I'll just read `process.env.X` for now and tighten it later" gets a chance to take root.
- **How it works:** Zod schemas are composed by section (`appEnvSchema`, `logEnvSchema`) and `.merge()`d into the root `envSchema`. `validateEnv(raw)` runs `safeParse` and either returns the typed result or throws a formatted error listing every issue (path + message). `@nestjs/config`'s `forRoot({ validate: validateEnv })` invokes this on module init, so a missing/malformed value crashes the process before any controller listens. `AppConfigService` wraps `ConfigService<Env, true>` so the rest of the app reads `config.port` (typed `number`) rather than `config.get('APP_PORT')` with a `string | undefined` return. The module is `@Global()` so any feature module can inject `AppConfigService` without re-importing.
- **Key concepts:** Twelve-factor config (env-based, validated on boot), Zod section composition (`merge`), `ConfigService<T, true>` "was-validated" generic for typed inference, NestJS global modules, fail-fast startup.
- **Best practices applied:** Validation as a pure function (`validateEnv` is testable without booting Nest). Section schemas composed at the top — each future Phase 2 item adds one section without touching the others. Env var names follow `NamingConventions.md` §8 prefix rules (`APP_`, `LOG_`). Defaults are explicit in the schema, not scattered through call sites. `.env.example` documents every variable in one place so new contributors don't grep for `process.env`. Test environment is configured in `vitest.config.ts` (`test.env`), keeping individual test files clean of env juggling.
- **Mistakes to avoid:** Reading `process.env` directly in any module past this point — once one place does it, the typed schema stops being the source of truth. Putting secret values in `.env.example` (the file is committed; it documents shape, not values). Letting Nest's `ConfigService` be injected directly into feature code instead of `AppConfigService` — without the wrapper, every consumer re-implements the `get('KEY') ?? throw` dance. Adding the not-yet-required env values (DB, Redis, JWT, Telebirr) to the schema speculatively now — every entry in the schema is a startup requirement, so adding a field before its consumer exists means dev environments break for no reason. Each later Phase 2 item adds its section as part of its own checklist work.

## Set up secrets management

- **What was done:** Logged the platform-agnostic secrets management policy in `DECISIONS.md` (dev `.env` gitignored, prod/staging via the deployment platform's native secret store, no secrets in code/images/logs/`.env.example`, per-env credentials, rotation before launch). Added a `SECRET_REGISTRY` to `src/config/env.schema.ts` — an array of `{ key, devPlaceholders }` entries — and an `assertNoDevSecretPlaceholders(env, registry?)` guard that runs after Zod parses in `validateEnv`: if `APP_NODE_ENV` is `staging` or `production` and any registered secret still holds one of its known placeholder strings, startup throws with a per-variable error pointing to which secret to rotate. The registry is empty right now (no secrets in the schema yet); each later Phase 2 item that introduces a secret (`DB_PRIMARY_URL`, `JWT_SIGNING_KEY`, `TELEBIRR_APP_SECRET`, `TELEBIRR_RSA_PRIVATE_KEY`, storage credentials) appends its own entry. Documented the policy at the top of `.env.example`. Tests cover: dev/test skip the check, prod/staging throw on placeholder match, prod passes when no placeholder match, empty registry passes (16/16 total tests green).
- **Why:** The dependency lines on this checklist item say "Deployment platform, CI/CD secret injection, access control policy" — but the deployment platform hasn't been narrowed past "containerized, managed Postgres+Redis" yet. Rather than block this Critical-priority item on a platform pick, the policy is written to be platform-agnostic: the *discipline* (where secrets live, what cannot happen, when they rotate) is the part this item commits to. The actual store binds at the deployment-architecture item. The startup guard exists *now* so when later items add the first real secret, the placeholder check is already in place — no later "we should add this check" item that gets dropped.
- **How it works:** `validateEnv` runs Zod parse → on success it calls `assertNoDevSecretPlaceholders(parsed)` → that function early-returns unless `APP_NODE_ENV` is `production`/`staging`, then iterates `SECRET_REGISTRY` comparing each entry's current value against its `devPlaceholders` list. Any matches collect into an offenders array; non-empty offenders throw a formatted error. The guard accepts an optional second argument (`registry`), which lets tests pass synthetic registries without mutating the module-level constant — the public default still uses the real registry.
- **Key concepts:** Secret rotation, defense in depth (Zod validation + placeholder guard + log redaction + gitignored `.env`), platform-native secret stores vs. dev `.env`, registry pattern with dependency-injectable defaults for testability.
- **Best practices applied:** No secrets in `.env.example` — the file is committed and documents shape, not values. Per-environment credentials (cross-env reuse is explicitly forbidden in DECISIONS.md). The guard runs *after* Zod parse so type-safety is intact when the placeholder check executes. Empty registry is the safe default — items must explicitly opt in by appending, which makes "is this a secret?" an explicit decision per variable rather than an inferred one.
- **Mistakes to avoid:** Committing a `.env` with real values (the rule is `.env` is gitignored, only `.env.example` is committed — and that contains no real values). Re-introducing direct `process.env.X` reads after the config layer was put in — every later item must add its variable to the schema and, if applicable, to `SECRET_REGISTRY`. Putting the placeholder guard in the deployment-platform layer instead of the config layer — that would only catch the problem at the wrong process boundary. Treating "the placeholder isn't in the registry" as proof the value is fine — the registry is best-effort against *known* placeholders; the real defense is per-environment rotation, not the guard.

## Create fully separated environments

- **What was done:** Logged a dedicated `Environment separation` decision in DECISIONS.md consolidating the cross-env rules (separate Postgres, Redis, storage, queues, Telebirr credentials, JWT signing keys, and admin accounts per environment; no cross-env reuse of any resource; production data never copied into lower environments without project-owner approval + PII/payment masking). Added a second registry to `env.schema.ts` — `DEV_INFRA_REGISTRY` with entries shaped `{ key, patterns: RegExp[] }` — and an `assertNoDevInfraValues(env, registry?)` guard wired into `validateEnv` after the placeholder check. In `staging`/`production` the guard tests each registered variable against its patterns (e.g. `/localhost/i`, `/127\.0\.0\.1/`, `/host\.docker\.internal/i`) and refuses startup if any matches. The registry is empty until later items add connection-string variables (`DB_PRIMARY_URL`, `REDIS_URL`, `STORAGE_ENDPOINT`, etc.). Updated `.env.example` with an environment-separation section. 21/21 tests pass.
- **Why:** The actual provisioning of separate databases/buckets/queues happens in the deployment-architecture, database, storage, and queue items — none of which exist yet. The deliverable for this Critical-priority item is the *policy* (recorded in DECISIONS.md so every later item that provisions a resource knows it must be per-env) and the *in-process guard* (a structural safety net that refuses to start with a localhost DB URL pointed at production). The guard is not a substitute for separate infrastructure — it's a last line of defense against a copy-paste mistake into the wrong secret store.
- **How it works:** `validateEnv` now runs three checks after Zod parse: (1) Zod schema validation, (2) `assertNoDevSecretPlaceholders` (string-equality match against known placeholder values), (3) `assertNoDevInfraValues` (regex match against dev-infrastructure patterns). The two registries are intentionally separate because they target different patterns: `SECRET_REGISTRY` is for short magic strings like `"changeme"`, `DEV_INFRA_REGISTRY` is for substrings like `"localhost"` inside larger connection URLs. Both guards skip in `development`/`test` and only enforce in `staging`/`production`. Both accept an optional registry argument so tests pass synthetic registries without touching the module-level constants.
- **Key concepts:** Defense in depth at the config boundary, environment isolation as both infrastructure policy and in-process check, regex vs. exact-string validation for different threat shapes.
- **Best practices applied:** Two narrowly-scoped registries instead of one generic "validator list" — each one's contract is obvious from its name. Patterns are case-insensitive where the value is hostname-like, exact where it's a fixed token. Guards run after Zod parse so the input is typed. Both guards have the same shape (`(env, registry?) => void`) so the wiring in `validateEnv` is symmetric and a third guard would follow the same pattern.
- **Mistakes to avoid:** Conflating "infrastructure separation" with "credential rotation" — they are independent and both required. Skipping the masking rule for prod→staging copies "just this once" for debugging — the rule exists precisely to stop that scenario. Treating an empty `DEV_INFRA_REGISTRY` as "the guard isn't doing anything" — it's the framework that makes the later DB/Redis/Storage items a one-line addition rather than a per-item bespoke check. Forgetting that the `test` environment also needs isolation — disposable per-run resources (e.g. test-scoped database, isolated Redis logical DB) are required, not "use the dev DB but clean up after."

## Provision PostgreSQL with geospatial support

- **What was done:** Logged a `PostgreSQL + PostGIS provisioning` decision in DECISIONS.md pinning the choice to Postgres 16 + PostGIS, with managed-provider PostGIS support as a hard requirement for staging/production. Added a `docker-compose.yml` at the repo root running the `postgis/postgis:16-3.4` image for local dev (`parks_dev` database, `parks/parks_dev` credentials, named volume for persistence, health check). Added `DB_PRIMARY_URL` to the Zod env schema as a required Postgres connection string (refines on `postgres://` / `postgresql://` scheme). Registered `DB_PRIMARY_URL` in `SECRET_REGISTRY` (matching three known dev URL strings exactly) AND `DEV_INFRA_REGISTRY` (matching `localhost` / `127.0.0.1` / `host.docker.internal` substrings) — belt and suspenders against a copy-paste landing in production. Added a `DatabaseConfigService` exposing `primaryUrl`, registered globally via `ConfigModule`. Updated `.env.example` and Vitest's env block. Refactored the env-schema test suite to spread from a `baseEnv` constant (everything stays compact even as the schema grows). 26/26 tests pass; `docker compose config` parses the compose file cleanly.
- **Why:** PostGIS is the standard for spatial queries on Postgres and the facility-discovery API depends on bounded-box / nearest-N queries that don't scale on naive lat/lng filtering. Locking in PostGIS 16 (specifically the `postgis/postgis:16-3.4` image) now means every later schema migration can `CREATE EXTENSION IF NOT EXISTS postgis` without later debate. Providing the compose file in the repo eliminates a friction point for any contributor: one command (`docker compose up -d`) gives them the same DB the rest of the team has. The two-registry registration of `DB_PRIMARY_URL` (placeholder + dev-infra) means the first secret to flow through the guards is also the one with the most realistic copy-paste threat — the structure is no longer theoretical.
- **How it works:** Local development reads `DB_PRIMARY_URL` from a gitignored `.env` (copied from `.env.example`); the compose stack publishes Postgres on `localhost:5432` so the dev URL matches. Tests get a different DB URL via `vitest.config.ts → test.env.DB_PRIMARY_URL` (`parks_test` database). Staging/production receive `DB_PRIMARY_URL` via the deployment platform's secret store. At startup, the config layer (1) Zod-validates the URL scheme, (2) refuses to boot in staging/production if the value equals a known dev URL placeholder (`SECRET_REGISTRY`), (3) refuses to boot in staging/production if the URL contains `localhost`/`127.0.0.1`/`host.docker.internal` (`DEV_INFRA_REGISTRY`). The actual `CREATE EXTENSION` migration runs as part of the next checklist item (ORM/migration setup).
- **Key concepts:** PostGIS as the spatial extension, managed-Postgres provisioning constraint (provider must support PostGIS), Docker Compose for local dev parity, Zod URL refinement, two-layer placeholder + dev-infra guards, named Docker volumes for persistence.
- **Best practices applied:** Pinned image tag (`postgis/postgis:16-3.4`) rather than `:latest` — version drift in local dev produces silent bugs. PostGIS image instead of vanilla `postgres:16` + extension scripting — saves a setup step and matches what the managed-provider plan must offer. Health check defined on the compose service so future items that depend on it can wait correctly. Separate `DatabaseConfigService` rather than dumping DB getters onto `AppConfigService` — keeps the typed-accessor surface clean as the config schema grows. Test env data set in `vitest.config.ts` (`test.env`) so individual specs don't repeat the boilerplate. `baseEnv` constant in the spec file means a future schema field adds one line to one constant instead of touching every test case.
- **Mistakes to avoid:** Using vanilla `postgres:16` and trying to `CREATE EXTENSION postgis;` later — the extension binaries aren't in that image, the migration fails, and you discover it on first run. Reusing `parks_dev` credentials anywhere outside local dev — that's exactly what the placeholder + dev-infra guards refuse. Reading `DB_PRIMARY_URL` from `process.env` directly later (bypassing `DatabaseConfigService`) — the typed accessor is the contract, and bypassing it loses startup validation. Treating "the compose file works on my machine" as proof staging/production will work — staging/production require a managed-provider PostGIS plan, which is its own dependency on the deployment-architecture item.

## Set up ORM or query builder with migration discipline

- **What was done:** Installed Drizzle ORM (`drizzle-orm ^0.45.2`) and Drizzle Kit (`drizzle-kit ^0.31.10`). Created a `@Global()` `DatabaseModule` that provides two tokens: `DRIZZLE_POOL` (a `pg.Pool` configured with connection string from `DatabaseConfigService.primaryUrl`, max 20 connections, 30s idle timeout, 10s connection timeout) and `DRIZZLE_CLIENT` (a `NodePgDatabase<typeof schema>` wrapping that pool via `drizzle(pool, { schema })`). The module imports `ConfigModule` for DI access to `DatabaseConfigService` and implements `OnApplicationShutdown` to close the pool gracefully. Both tokens are exported for injection into domain modules. Created `src/database/drizzle.schema.ts` as an empty schema placeholder (domain tables come in Phase 3). Created the first migration `drizzle/0000_enable_postgis.sql` with `CREATE EXTENSION IF NOT EXISTS postgis;` plus proper Drizzle Kit metadata (`drizzle/meta/_journal.json` v7, `drizzle/meta/0000_snapshot.json` with empty schema). Created `src/migrate.ts` as a standalone migration runner using `drizzle-orm/node-postgres/migrator` that reads `DB_PRIMARY_URL` from env (loaded via `dotenv` from `.env` if present), validates the connection string format, applies migrations from `./drizzle`, and exits. Created `drizzle.config.ts` for Drizzle Kit CLI with the same env-based config and explicit error messages for missing/malformed URLs. Added npm scripts: `db:generate` (drizzle-kit generate), `db:migrate` (runs migrate.ts via tsx), and `db:studio` (drizzle-kit studio). Fixed `@types/pg` moved from `dependencies` to `devDependencies` (type-only package). Installed `tsx` as a devDependency (replaces the never-installed `ts-node`). Registered `DatabaseModule` in `AppModule.imports`. Updated tests: `drizzle.module.spec.ts` now tests both module compilation without a live DB and pool shutdown via `onApplicationShutdown` mock; `app.module.spec.ts` overrides `DRIZZLE_POOL` alongside `DRIZZLE_CLIENT`. 28/28 tests pass (was 27 — one new shutdown test).
- **Why:** Drizzle was already chosen in DECISIONS.md (stays close to SQL, supports atomic conditional updates and exclusion constraints natively). The PostGIS extension must be enabled via SQL before any spatial column or index is created — the first migration does exactly that. Separating pool and client as two DI tokens lets lifecycle management (pool.end() on shutdown) live cleanly in the module class without requiring domain code to know about pool management. Using `tsx` instead of `ts-node` gives faster TypeScript execution and better Node compatibility. The ".env loading via dotenv" approach in the standalone migration runner keeps the CLI tooling decoupled from NestJS's config layer while still supporting local development without manual env setup.
- **How it works:** At NestJS startup, `DatabaseModule` creates a `pg.Pool` via factory (injecting `DatabaseConfigService`) and wraps it with `drizzle(pool, { schema })` to produce the `DRIZZLE_CLIENT`. Any domain module injects `@Inject(DRIZZLE_CLIENT)` to get the typed query builder. On shutdown, `DatabaseModule.onApplicationShutdown()` calls `this.pool.end()` to drain idle connections and close the pool. The migration runner (`src/migrate.ts`) is a standalone script — it uses `dotenv` to optionally load `.env`, reads `DB_PRIMARY_URL`, validates the postgres:// scheme, creates a single-connection pool, calls `migrate()` from drizzle-orm's node-postgres migrator, then closes the pool. Drizzle Kit's generate command (`npm run db:generate`) diffs the schema file against the snapshot in `drizzle/meta/` and writes new migration SQL to `drizzle/`. Since the schema is empty, generate reports "0 tables, nothing to migrate". The `_journal.json` tracks which migrations have been applied — the migrator skips already-applied entries.
- **Key concepts:** Drizzle ORM as a query builder (not a full ORM — no auto-loading, no identity map), Drizzle Kit for migration generation, `node-postgres` pool lifecycle, NestJS `OnApplicationShutdown` lifecycle hook, standalone migration runner pattern, `dotenv` for CLI env loading, Drizzle Kit `_journal.json` + `snapshot.json` metadata format.
- **Best practices applied:** Pool lifecycle managed at the module level (clean shutdown without resource leaks). Two separate DI tokens (`DRIZZLE_POOL` and `DRIZZLE_CLIENT`) keep concerns separated — the pool is an infrastructure detail, the client is the application-facing interface. Tests override both tokens with lightweight stubs (no real Postgres needed). Migration runner validates connection string format before attempting connection, giving clear error messages for common mistakes. CLI tooling config (`drizzle.config.ts`) provides explicit error messages instead of unchecked `!` assertions. `dotenv` loading in the standalone script is optional and silent on missing `.env` — follows the same pattern `@nestjs/config` uses. `@types/pg` correctly placed in devDependencies (type-only, no runtime value). `tsx` installed explicitly rather than relying on npx or a transitive binary.
- **Mistakes to avoid:** Forgetting to override both `DRIZZLE_POOL` and `DRIZZLE_CLIENT` in tests — the pool factory runs during module compilation and tries to create a real database connection. Omitting `ConfigModule` from `DatabaseModule.imports` — the factory injects `DatabaseConfigService` which is provided by `ConfigModule`, so the module must import it or the DI context fails. Hardcoding `process.env.DB_PRIMARY_URL!` without validation — a missing URL crashes with an unhelpful cryptic error instead of a clear "set DB_PRIMARY_URL" message. Relying on `ts-node` without adding it to dependencies — the script silently fails when someone runs npm install fresh. Leaving `@types/pg` in runtime dependencies — it adds noise to the dependency tree and is not used at runtime. Creating a custom SQL migration without updating `_journal.json` — the Drizzle migrator won't pick it up. Forgetting `--env-file` or `.env` loading for the migration runner — developers expect `npm run db:migrate` to find their local `.env`.

## Create API documentation pipeline

- **What was done:** Added `@nestjs/swagger` (^11.4.4) to dependencies. Created `src/docs/docs.module.ts` and `src/docs/docs.service.ts` that configure OpenAPI document metadata (title: "Public Recreation Facility Management Platform", version: 0.1.0, description covering all three client types). Added a placeholder BearerAuth security scheme marked as "not yet implemented" — this matches the roadmap without pretending auth exists. Created `src/docs/generate-openapi.ts`, a standalone script that bootstraps AppModule via `@nestjs/testing` with stubbed database providers (no live DB needed), generates the OpenAPI document via SwaggerModule, and writes `dist/openapi.json`. Added npm script `docs:generate` that sets the required env vars (`APP_NODE_ENV`, `DB_PRIMARY_URL`, `APP_ENABLE_DOCS`) before running. Added `APP_ENABLE_DOCS` (default: `false`) to the Zod env schema and `AppConfigService` getter. Updated `main.ts` to call `docsService.setupSwaggerUi(app)` — which mounts Swagger UI at `/docs` and JSON at `/docs-json` in all environments except production (production is gated unless `APP_ENABLE_DOCS=true` is explicitly set). Registered `DocsModule` in `AppModule.imports`. Updated `baseEnv` in the env-schema spec and all test env objects to include `APP_ENABLE_DOCS: false`. Added 9 new tests for `DocsService` (module compilation, path constants, enable/disable logic across environments). 42/42 tests pass, typecheck green, build green, `npm run docs:generate` produces a valid OpenAPI 3.0 JSON with 0 paths (expected — no controllers exist yet), 2 components (schemas + securitySchemes), and 1 auth scheme (bearer placeholder).
- **Why:** Without a documented API contract, mobile and staff app developers have no stable reference to build against and no way to catch breaking changes before they reach production. Setting this up in Phase 2 — before any Phase 5 API is written — means every future controller decorated with NestJS decorators automatically contributes to the OpenAPI document, and the contract generation is a `npm run docs:generate` away with no running server or database required. Gating Swagger UI in production by default is deliberate: interactive docs are a development/QA tool that expose endpoint shapes, and in a government system they should not be available on the production domain unless an operator explicitly enables them.
- **How it works:** `DocsService.createOpenApiDocument(app)` builds a `DocumentBuilder` with title, description, version, and a bearer-auth placeholder, then calls `SwaggerModule.createDocument()` to produce an OpenAPI 3.0 object reflecting all registered controllers and their DTOs. `DocsService.setupSwaggerUi(app)` gates on `shouldEnableDocs()` — which returns `true` if `APP_NODE_ENV` is not production OR if `APP_ENABLE_DOCS` is explicitly `true` — then calls `SwaggerModule.setup()` to mount the Swagger UI and JSON endpoints on the Express app. The generator script (`generate-openapi.ts`) overrides `DRIZZLE_POOL` and `DRIZZLE_CLIENT` with stubs in a `Test.createTestingModule` context, so it never opens a database connection — the config validation is satisfied by env vars set in the npm script prefix. The generated JSON is written to `dist/openapi.json` for CI artifact or client-consumption workflows. `APP_ENABLE_DOCS` uses a union of Zod literal coercions that accept only `true`, `false`, `'true'`, `'false'`, `'1'`, `'0'` — malformed values like `'yes'`, `'treu'`, `''`, or `2` are rejected with a clear error (unlike the original `z.string().or(z.boolean()).or(z.number())` approach which silently coerced `'yes'` to `false`).
- **Key concepts:** OpenAPI 3.0 specification, Swagger UI as a documentation viewer, `@nestjs/swagger` decorators for automatic schema generation, `SwaggerModule.createDocument()` vs `SwaggerModule.setup()`, standalone document generation with stubbed infrastructure, boolean coercion edge cases in Zod.
- **Best practices applied:** Interactive docs disabled in production by default — an explicit `APP_ENABLE_DOCS=true` opt-in is required. The bearer auth scheme is declared with a description that explicitly says "not yet implemented" — this prevents API consumers from assuming JWT auth works before Phase 4/5 deliver it. The generator script uses `Test.createTestingModule` with overridden DB providers so it never needs a live database and never leaves connections dangling (`.close()` called on success). The generator script writes to `dist/` (already gitignored) so generated artifacts don't pollute the source tree. The version string `0.1.0` reflects pre-Phase 5 status — no public endpoints exist yet. The `docs:generate` npm script sets env vars via shell prefix so the module imports find them at load time.
- **Mistakes to avoid:** Using `z.coerce.boolean()` naively — the string `"false"` is coerced to `true` because `Boolean("false") === true` in JavaScript. This broke one test and had to be replaced with a custom string/boolean/number transform that only accepts `"true"` and `"1"` as truthy. Forgetting to register `DocsModule` in `AppModule.imports` — without this, `DocsService` is not available in the DI container and `main.ts` throws at startup. Making the generator require a live database — that adds a setup dependency for a documentation build step and means CI needs a Postgres instance just to generate an API contract. ## Create standard request validation layer

- **What was done:** Created `src/common/validation/` with five files that together deliver a global, default-on validation layer wired into NestJS via `APP_PIPE` and `APP_FILTER`:
  - `create-zod-dto.util.ts` — `createZodDto(schema)` returns a class (`ZodDtoStatic`) carrying the Zod schema as a static `zodSchema` property. Route handlers type their body parameters against these classes; NestJS's DI resolves the class as the param metatype, which the pipe reads without any per-route annotation.
  - `request-validation.exception.ts` — `RequestValidationException` (extends `Error`) with `code: 'VALIDATION_FAILED'` and `details: Record<string, string[]>` (field-path-keyed error arrays). Keeps the domain exception decoupled from HTTP — the filter is responsible for rendering it as a response.
  - `zod-validation.pipe.ts` — `ZodValidationPipe` registered globally as `APP_PIPE`. It reads `zodSchema` off the route param metatype, runs `safeParse`, passes through unchanged when no schema is present (so non-DTO params are unaffected), and throws `RequestValidationException` on failure. Zod's `error.flatten()` output is reshaped: field issues go into `details[fieldPath]`, root issues (schema-level failures not tied to a field) go into `details['_root']`.
  - `validation-exception.filter.ts` — `ValidationExceptionFilter` registered globally as `APP_FILTER` with `@Catch(RequestValidationException)`. Renders HTTP 400 with the canonical envelope `{ error: { code, message, details, correlationId } }`. The `correlationId` field is the exported constant `CORRELATION_ID_PLACEHOLDER = 'not-yet-wired'` — a deliberate stub that will be replaced when the structured-logging item wires real per-request correlation IDs. Response is typed via a local `HttpResponseLike` interface so the filter has no dependency on `@types/express`.

  A `DECISIONS.md` entry ("Class-suffix naming for NestJS framework primitives") was added this item, extending `NamingConventions.md` §3: NestJS Pipe/Guard/Filter/Interceptor/Middleware classes use their idiomatic NestJS suffix (`Pipe`, `Guard`, `Filter`, `Interceptor`, `Middleware`) rather than the generic `Service` suffix. No new npm dependency was introduced.

- **Why:** Input validation at the API boundary is OWASP A03 (Injection) and A04 (Insecure Design) territory. Every Phase 5 API endpoint takes external data; if the validation layer were implemented per-route with a per-route pipe, a single forgotten annotation would leave that endpoint entirely unvalidated — a "fails open" posture. Registering the pipe and filter globally via `APP_PIPE`/`APP_FILTER` makes validation default-on: a route only skips validation if it deliberately has no typed DTO parameter, not because the developer forgot to add the pipe. The canonical error envelope (`VALIDATION_FAILED`, field-level `details`) is committed now — before any real API exists — so every Phase 5 API returns the same shape on input failure without later rework. Owning the `correlationId` field as a placeholder also defers the structured-logging dependency without making the field optional in the contract.

- **How it works:** When NestJS processes a route handler, it inspects each parameter decorated with `@Body()`, `@Query()`, etc. via `Reflect.metadata`. The parameter's metatype (the TypeScript class) is passed to each registered pipe's `transform(value, metadata)` method. `ZodValidationPipe.transform` checks `metadata.metatype` for a static `zodSchema` property: if found, it calls `zodSchema.safeParse(value)`. On `success: false`, the pipe constructs a `RequestValidationException` from `error.flatten()`. On `success: true`, the pipe returns the parsed (and coerced or defaulted) value — not the raw input. If `zodSchema` is absent (a primitive param, a string, etc.), the pipe returns the raw value unchanged so existing non-DTO parameters work without modification. When `RequestValidationException` propagates, `ValidationExceptionFilter.catch` intercepts it, reads the request to extract a future correlation ID (currently the placeholder constant), and writes the 400 response using the locally-typed `HttpResponseLike` interface instead of importing from `express`.

- **Key concepts:** NestJS global enhancers (`APP_PIPE`, `APP_FILTER`) vs. per-route enhancers; `Reflect.metadata` and `emitDecoratorMetadata` for metatype resolution; Zod `safeParse` vs. `parse` (safe = no throw, returns discriminated union); `error.flatten()` producing `fieldErrors` (keyed by field path) and `formErrors` (root-level failures); `ZodDtoStatic` interface allowing the pipe to read `zodSchema` from any DTO class; `@Catch(RequestValidationException)` for type-safe exception filter binding; `HttpResponseLike` local interface as a `@types/express` substitute.

- **Best practices applied:** Global registration rather than per-route — validation is default-on and cannot be accidentally omitted from a new endpoint. `.strict()` schemas are used for all DTO schemas — unknown fields are REJECTED, not silently stripped. Rejecting unknown fields (rather than stripping) is the correct OWASP posture: stripping allows attackers to probe which fields the backend accepts; rejecting closes that information channel. The `VALIDATION_FAILED` error code is a string constant, not a free-text string — clients can reliably switch on it. The `correlationId` is a named constant (`CORRELATION_ID_PLACEHOLDER`) rather than an inline `'not-yet-wired'` string — this makes the placeholder easy to find and replace as a single refactor when the logging item ships. `emitDecoratorMetadata: true` is already set in `tsconfig.json`; the e2e test manually injects the DTO metatype so the test works under esbuild/Vitest (which does not emit decorator metadata). The filter uses a local `HttpResponseLike` interface rather than importing `@types/express` — keeping the dependency surface minimal.

- **Mistakes to avoid:** Registering the validation pipe per-route instead of globally — a single forgotten `@UsePipes(ZodValidationPipe)` means an endpoint accepts arbitrary unvalidated input. Using Zod's `.strip()` (or the default `.strip()` behavior on `z.object`) instead of `.strict()` — stripping unknown fields allows clients to probe which keys the backend reads without surfacing an error, which is an OWASP A04 concern. Throwing inside `transform` with a raw `Error` instead of `RequestValidationException` — the filter catches `RequestValidationException` specifically; a raw `Error` falls through to NestJS's default handler and produces a 500 with a generic message. Leaking Drizzle column names into `details` keys — the `details` field is client-facing; if the field-path key is the snake_case database column name (e.g., `facility_id`) rather than the camelCase DTO field name (e.g., `facilityId`), it exposes schema internals and creates a client API surface tied to the database layout. Omitting `emitDecoratorMetadata: true` from `tsconfig.json` — without it, `metadata.metatype` resolves to `Object` at runtime (not the DTO class), so the pipe never finds `zodSchema` and passes all values through unvalidated. Calling `zodSchema.parse(value)` instead of `safeParse` inside the pipe — `parse` throws a `ZodError`, which is not a `RequestValidationException`; the filter does not catch it, and NestJS renders it as a 500.
